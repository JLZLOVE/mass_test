注销功能逻辑设计说明书
模块：认证与授权 关联模块：JwtFilter、UserCacheHelper 版本：v1.0 | 2026-07-17
1. 功能概述
注销功能用于终止当前登录用户的会话。其核心机制为：删除 Redis 中该用户的登录缓存，使同一 Token 在后续请求中因缓存缺失而被 JwtFilter 拦截，返回 401。
2. 原理解析
2.1 认证链回顾
当前请求通过认证须满足三项条件：

格式合法、未过期、能提取 `username`
username 有效
`loadUserByUsername` 能找到对应用户

`user:v2:{username}` 键存在且可反序列化
注销即破坏第三条——删除缓存后，Token 即使合法也拿不到用户信息。

4. 处理流程
4.1 主流程
POST /login/logout 携带 Token 进入 Controller
        │
        ▼
从 SecurityContext 提取当前登录用户 → 取 username
        │
        ▼
构造 Redis Key：user:v2:{username}
        │
        ▼
执行 redisTemplate.delete(key)
        │
        ▼
记录操作日志（上下文仍可用）
        │
        ▼
返回 R.ok("已退出") → 请求结束
        │
        ▼
（框架自动清理 SecurityContext）
4.2 关键步骤说明
步骤
说明
**提取用户**
`SecurityContextHolder.getContext().getAuthentication()` → `principal` 为 `LoginUserDetails`。若为空或类型不符，视为已退出，直接返回成功。
**删除缓存**
调用 `UserCacheHelper.keyForUsername(username)` 构造键名，执行 `redisTemplate.delete(key)`。键不存在时 `delete` 返回 0，不影响业务语义。
**清空上下文**
**不手动调用** `clearContext()`。Spring Security 的 `SecurityContextPersistenceFilter` 会在请求完全结束后自动清理当前线程的 ThreadLocal。手动清理会导致后续响应处理组件（如 `ResponseBodyAdvice`、`@ControllerAdvice` 日志记录）在需要读取上下文时获取不到用户信息，引发异常。

6. 安全考量
6.1 "旧 Token 复活"现象
现象
原因
评估
用户注销后重新登录，此前持有的旧 Token 再次有效
Redis 缓存以 `username` 为键，重新登录写入新缓存；旧 Token 解析出的 username 与新缓存键匹配
**可接受**。校园社团管理系统安全等级有限；攻击需同时满足 Token 窃取 + 用户重新登录 + 1 小时窗口；Token 通过 HTTPS 承载。
6.2 与管理员禁用用户联动
管理员通过 /sys-user/toggleStatus 禁用用户时，已调用 UserCacheHelper.evictByUsernames 清理该用户的 Redis 缓存。两者作用于同一 Redis 键，语义一致，无冲突。
