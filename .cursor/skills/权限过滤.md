### 一、业务规则（五级可见范围）

| 等级  | 身份                | 可见范围                        |
| :---- | :------------------ | :------------------------------ |
| 1     | 超级管理员（校长）  | **全部**                        |
| **2** | 管理员（老师/导员） | **所有学生**（不含教师/管理员） |
| **3** | 社长                | **本社团**                      |
| **4** | 部长                | **本部门**                      |
| 5     | 普通学生            | **仅自己**                      |

------

### 二、技术方案（零侵入，三处改动）

1. **改造 `LoginUserDetails`（实现 `UserDetails`）**
   - 组合 `SysUser` + `有效等级` + `社团ID` + `部门ID`。
   - 尽量少建新类.
2. **改造 `UserDetailServiceImpl`（一次性加载）**
   - 查用户 → 查角色 → 取最小 `role_level` → 根据等级查 `scope_id`（从 `sys_user_role` 取） → 全部塞进 `LoginUserDetails` 返回。
   - **目的**：从此处就带齐所有数据，避免后续二次查询。

------

### 三、约束与兼容

- **`JwtFilter` 只做令牌校验**，严禁在此处查 DB 查等级（只调 `loadUserByUsername` 或读 Redis 缓存）。
- **Redis 缓存**：改为缓存整个 `LoginUserDetails`（原缓存 `Collection` 的方式可废弃或兼容过渡）。
- **现有 `BaseQuery` 不变**：切面注入条件（`setClubId`、`setUserId` 等逻辑保留，但数据源换成 `LoginUserDetails`）。
- **不修改** `SecurityConfig`、`LoginController`、现有 Entity 表结构。
