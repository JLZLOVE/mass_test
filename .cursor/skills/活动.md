## 创建申请

接口：创建申请

入参：社团名称、挂靠学院、社团类别、简介、拟定社长ID、最大人数

逻辑：

1. 当前用户必须持有指导老师角色
2. 拟定社长必须存在且未被禁用
3. 同一学院下社团名称不得重复（含已解散的）
4. 生成申请编号
5. 插入申请记录，状态=待学院审批

## 解散申请

接口：解散申请

入参：社团ID、解散原因

逻辑：

1. 当前用户必须是指定社团的指导老师
2. 社团必须处于正常状态
3. 社团无进行中活动（草稿、待审、审批中）
4. 插入申请记录，状态=待学院审批

## 学院审批

接口：学院审批

入参：申请ID、通过/驳回、意见

逻辑：

1. 当前用户必须等于挂靠学院的dean_id
2. 申请状态必须为待学院审批
3. 当前用户不能是申请人
4. 通过→状态变为学院已通过
5. 驳回→状态变为已驳回，记录原因

## 校级审批

接口：校级审批

入参：申请ID、通过/驳回、意见

逻辑：

1. 当前用户必须持有ADMIN角色
2. 申请状态必须为学院已通过
3. 当前用户不能是申请人
4. 通过→执行激活社团，状态变为已通过
5. 驳回→状态变为已驳回

## 激活社团（创建通过时自动触发）

逻辑：

1. 插入社团记录，指导老师=申请人
2. 生成社团编号
3. 给拟定社长绑定CLUB_LEADER角色，scope_type=2，scope_id=新社团ID

## 合议发起

接口：发起合议

入参：社团ID、原因

逻辑：

1. 当前用户必须持有SUPER_ADMIN角色
2. 社团必须处于正常状态
3. 当前用户在该社团挂靠学院的数据范围内（超管全校管全校，分学院管分学院）
4. 该社团无进行中的合议
5. 插入合议记录，状态=合议中，签名列表为空

## 合议签字

接口：合议签字

入参：合议ID

逻辑：

1. 合议记录存在且状态为合议中
2. 签字人必须持有SUPER_ADMIN或ADMIN角色
3. 签字人在该社团挂靠学院的数据范围内
4. 同一人不能重复签字
5. 记录签名
6. 每次签名后检查是否达成条件：
   - 条件A：至少1个超管+至少2个ADMIN → 通过
   - 条件B：至少3个超管 → 通过
   - 否则继续等待
7. 达成条件→执行解散，合议状态变为已通过

## 执行解散（校级审批通过或合议达成时自动触发）

逻辑：

1. 社团状态变为已解散
2. 删除该社团所有部门
3. 清理活动：仅保留已结束且已通过的，其余全部删除
4. 移除该社团关联的CLUB_LEADER角色
5. 移除该社团所有部门关联的DEPT_LEADER角色

## 附加规则

1. 超管分学院管理：超管角色data_scope=1时可操作全校；否则仅可操作scope_type=1且scope_id匹配的学院
2. 合议签字人范围：签字人必须在该社团挂靠学院的数据范围内
3. 申请列表分页查询：支持按申请人、状态、类型、时间范围筛选
4. 详情查询：返回申请完整信息及当前审批人
