## 活动审批模块 — 核心逻辑

### 一、现有表确认

`activity_apply` 当前状态：
- `approve_status`: 1草稿 2待审批 3审批中 4已通过 5已驳回 6已取消
- `current_approve_step`: 当前执行到的步骤序号

`activity_approve_flow`：存储每一步审批记录（`step` 序号，`approve_role_id`，`approve_user_id`，`approve_result`，`approve_opinion`，`approve_time`）

### 二、新增字段/表

| 变更                            | 说明                                          |
| ------------------------------- | --------------------------------------------- |
| `activity_apply.activity_level` | TINYINT，1=院级 2=校级，发起时必填            |
| `activity_apply.version`        | INT 默认0，乐观锁                             |
| `activity_apply_history`        | 变更前快照表（字段同主表 + `history_reason`） |

### 三、发起申请

| 校验项     | 规则                                                         |
| ---------- | ------------------------------------------------------------ |
| 发起人角色 | 部长/社长/指导老师/学院书记/ADMIN                            |
| 活动级别   | 必填，发起人手动选择                                         |
| 附件       | 支持上传（本地存储，预留OSS）                                |
| 审批链生成 | 根据发起人角色+活动级别动态生成步骤，写入 `activity_approve_flow` |
| 初始状态   | `approve_status=2`（待审批），`current_approve_step=1`       |

### 四、审批链路径

| 发起人         | 第1级                  | 第2级    | 第3级    | 终点(院级) | 终点(校级) |
| -------------- | ---------------------- | -------- | -------- | ---------- | ---------- |
| 部长           | 所在社团社长           | 指导老师 | 学院书记 | 学院书记   | 校书记     |
| 社长           | 指导老师               | 学院书记 | —        | 学院书记   | 校书记     |
| 指导老师       | 学院书记               | —        | —        | 学院书记   | 校书记     |
| 学院书记/ADMIN | 指导老师（确认可行性） | 学院书记 | —        | 学院书记   | 校书记     |

### 五、审批人查找规则

| 角色     | 查找方式                                                     |
| -------- | ------------------------------------------------------------ |
| 社长     | `sys_user_role` 中 `role_code=社长` + `scope_type=2` + `scope_id=社团ID` |
| 指导老师 | `sys_club.advisor_id`                                        |
| 学院书记 | `sys_college.dean_id`（活动挂靠社团所属学院）                |
| 校书记   | `sys_role` 中 `role_code=SUPER_ADMIN` 的用户                 |

### 六、审批流转

| 操作     | 规则                                                         |
| -------- | ------------------------------------------------------------ |
| 通过     | 必须填写意见（"阅"有效）→ 写入当前步骤记录 → 无下一步则状态变为4（已通过），否则推进到下一步 |
| 驳回     | 必须填写原因 → 状态变为5（已驳回），流程终止                 |
| 级别调整 | 仅指导老师在审批时可将活动级别从1调整为2，或从2调整为1，**调整一次后锁定** |

### 七、变更申请（时间/地点）

| 规则项   | 说明                                                         |
| -------- | ------------------------------------------------------------ |
| 触发     | 已通过的活动（`approve_status=4`）可发起变更                 |
| 前置条件 | 原活动未被取消                                               |
| 审批路径 | 发起人 → 社长 → 指导老师 → 学院书记（逐级确认，不达校级）    |
| 变更内容 | 仅 `start_time`、`end_time`、`location`、`location_detail`   |
| 快照机制 | 变更申请发起时，**立即生成历史记录**（`activity_apply_history`），写入变更前的完整活动数据 |
| 变更生效 | 变更审批通过后，**主表更新**为新数据，历史记录保留供追溯     |
| 变更驳回 | 主表维持不变，历史记录保留（标记为"已驳回变更"）             |
| 状态     | 变更申请状态独立于原活动状态（原活动维持"已通过"，变更申请用 `status=7` 表示"变更审批中"） |

### 八、取消活动

| 规则项 | 说明                                     |
| ------ | ---------------------------------------- |
| 触发   | 发起人（原申请人）直接取消               |
| 审批   | 不需要审批                               |
| 动作   | `approve_status=6`（已取消），填取消原因 |
| 通知   | 通知已报名/已签到的参与人                |

### 九、超时机制

| 规则项   | 说明                                                         |
| -------- | ------------------------------------------------------------ |
| 触发条件 | 某步骤 `approve_time` 为 NULL，且当前时间 > 进入该步骤时间 + 3天 |
| 催办     | 系统推送催办通知给当前审批人                                 |
| 转交     | 超时 > 7天，自动转交该角色的上级（社长→指导老师，指导老师→学院书记，学院书记→校书记） |
| 实现     | 定时任务（Spring Scheduled），每小时扫描一次                 |

### 十、活动总结

| 规则项   | 说明                                                         |
| -------- | ------------------------------------------------------------ |
| 上传窗口 | 活动结束后 1-3 天内                                          |
| 内容     | 文字 + 附件（可不走审批）                                    |
| 存储     | 关联 `activity_apply` 记录（新增字段 `summary_content`、`summary_attachment`、`summary_upload_time`） |

### 十一、并发控制

| 规则项   | 说明                                                  |
| -------- | ----------------------------------------------------- |
| 机制     | 乐观锁（`version` 字段）                              |
| 触发     | 任何更新操作（审批、变更、取消）均需带 `version` 校验 |
| 失败处理 | 版本冲突时回滚，提示"数据已被修改，请刷新重试"        |

### 十二、编码约束

| 项           | 要求                                                         |
| ------------ | ------------------------------------------------------------ |
| 动态查找     | `ClubSecurityHelper` 中封装审批人查找                        |
| 审批链初始化 | 根据发起人角色+级别动态生成步骤，存入 `activity_approve_flow` |
| SQL          | Lambda 条件构造器                                            |
| 事务         | 发起、审批、变更、取消均加 `@Transactional`                  |
