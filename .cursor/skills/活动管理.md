# 活动管理模块 · 企业级交互设计规范 v1.0

> 文档性质：**业务逻辑说明书与交互行为准则**
> 适用范围：`club-admin-frontend` 一级菜单「活动管理」
> 设计核心：**身份驱动、状态机闭环、审批-签到-总结三段式分离**
> 更新日期：2026-07-24


## 1. 设计定位与核心概念

### 1.1 模块边界

活动是社团运营的核心载体。本模块覆盖一个活动从“构思”到“归档”的完整生命周期：

```
发起申请 → 多级审批 → 执行签到 → 提交总结 → 归档
     ↓           ↓           ↓           ↓
  草稿/提交   审批链流转   签到/签退   总结附件
```

### 1.2 四种用户身份（同一页面，不同视图）

活动模块的特殊性在于：**同一用户在不同活动中可能扮演不同身份，同一页面需根据身份动态调整内容与操作。**

| 身份类型 | 定义 | 在活动模块中的核心诉求 |
|---|---|---|
| **发起人** | 提交活动申请的用户（`apply_user_id`） | 跟踪审批进度、取消活动、提交总结 |
| **审批人** | 当前审批步骤的负责人（`approve_user_id`） | 通过/驳回、查看审批链、填写意见 |
| **签到执行人** | 活动参与人（签到记录中的 `user_id`） | 签到、签退、查看签到状态 |
| **查看人** | 有权限查看但无操作权限的用户 | 浏览活动信息、查看签到统计（只读） |

> **设计铁律**：活动模块的所有操作按钮**必须**基于当前用户在该活动中的「身份」与活动「状态」的笛卡尔积进行渲染，**禁止**仅凭 `effectiveLevel` 决定操作权限。详细映射见 §4。

### 1.3 活动状态机（前端核心数据模型）

活动状态是驱动 UI 渲染的核心变量。前端必须完整理解以下 8 种状态及其转换规则：

| 状态码 | 状态名 | 业务含义 | 前端视觉标识 |
|---|---|---|---|
| 1 | 草稿 | 用户保存但未提交 | 灰色标签「草稿」 |
| 2 | 待审批 | 已提交，等待第一步审批人处理 | 蓝色标签「待审批」 |
| 3 | 审批中 | 审批链已推进，等待后续步骤 | 橙色标签「审批中」 |
| 4 | 已通过 | 全部审批完成，可执行签到 | 绿色标签「已通过」 |
| 5 | 已驳回 | 审批被驳回，流程终止 | 红色标签「已驳回」 |
| 6 | 已取消 | 发起人主动取消 | 灰色标签「已取消」 |
| 7 | 变更审批中 | 已通过的活动发起变更，等待审批 | 紫色标签「变更审批中」 |
| 8 | 已封锁 | 防篡改机制触发，不可操作 | 暗红标签「已封锁」 |

**状态转换规则（前端不需校验，但需理解展示逻辑）：**

```
草稿(1) → 提交 → 待审批(2)
待审批(2) → 第一步审批 → 审批中(3)（若有多步）
审批中(3) → 最后一步审批 → 已通过(4)
已通过(4) → 发起变更 → 变更审批中(7)
变更审批中(7) → 变更通过 → 已通过(4)（数据更新）
变更审批中(7) → 变更驳回 → 已通过(4)（数据不变）
任意非草稿/已取消/已驳回状态 → 发起人取消 → 已取消(6)
已通过(4) → 活动结束后3天未提交总结 →（无状态变化，仅预警）
任意状态 → 防篡改触发 → 已封锁(8)（不可逆）
```


## 2. 用户分层与权限矩阵

### 2.1 页面级准入控制（路由层）

| 用户等级 | 菜单「活动管理」 | 数据范围 |
|---|---|---|
| Level 0/1（超管/院长/指导老师） | ✅ 可见 | 全平台活动（按管辖范围过滤） |
| Level 2（社长） | ✅ 可见 | 本社团下所有活动 |
| Level 3（部长） | ✅ 可见（`minLevel=3`） | **本部门挂靠社团下所有活动（仅只读）** |
| Level 4（普通学生） | ❌ 菜单不可见 | 通过工作台「我的活动」卡片跳转至个人活动列表 |

> **特别注意**：Level 3（部长）**仅有查看权限**，无权发起、审批、变更、取消或总结任何活动。其进入活动管理的唯一目的是**了解本部门相关的活动安排**，用于配合执行。

### 2.2 操作级权限矩阵（身份 × 状态）

> 这是活动模块最核心的设计表。前端必须根据「当前用户在该活动中的身份」+「活动状态」的交集决定操作按钮的显隐。

| 操作 | 发起人 | 审批人 | 签到执行人 | 查看人（部长等） |
|---|---|---|---|---|
| **查看详情** | ✅ | ✅ | ✅ | ✅ |
| **编辑草稿** | ✅（仅状态=1） | ❌ | ❌ | ❌ |
| **提交申请** | ✅（仅状态=1） | ❌ | ❌ | ❌ |
| **审批通过/驳回** | ❌ | ✅（仅当为当前步骤审批人） | ❌ | ❌ |
| **变更申请** | ✅（仅状态=4） | ❌ | ❌ | ❌ |
| **取消活动** | ✅（仅状态=2/3/4/7） | ❌ | ❌ | ❌ |
| **提交总结** | ✅（仅状态=4 且活动已结束且≤3天） | ❌ | ❌ | ❌ |
| **签到（自己）** | ❌ | ❌ | ✅（仅活动进行中且签到窗口内） | ❌ |
| **签退（自己）** | ❌ | ❌ | ✅（已签到且启用签退） | ❌ |
| **查看签到统计** | ✅（仅状态≥4） | ✅ | ✅ | ✅（仅状态≥4） |
| **补签审批** | ❌ | ✅（仅当为补签审批人） | ❌ | ❌ |

> **前端实现约束**：
> - 所有操作按钮必须通过**后端下发的权限标识集合**（如 `permissions: ['canApprove', 'canSign', ...]`）驱动渲染。
> - **严禁**在前端使用 `if (user.role === '社长' && activity.status === 4)` 这类硬编码判断。


## 3. 页面布局与功能区划分

### 3.1 整体结构

活动模块拆分为 **三个子页面**，职责分离：

| 子页面 | 路由 | 核心用途 | 进入方式 |
|---|---|---|---|
| **活动列表** | `/activity/apply` | 活动总览、筛选、快速签到入口 | 左侧菜单「活动管理」 |
| **活动详情** | `/activity/detail/:activityNo` | 完整信息、审批、变更、总结 | 列表页点击活动名称 |
| **签到执行** | `/activity/sign/:activityNo` | 定位签到、扫码签到、签退 | 列表页「去签到」按钮 / 扫码 |

### 3.2 活动列表页布局

```
+------------------------------------------------------------+

|                                                             |
| [数据行] WH20260724... | 迎新晚会 | 音乐社 | 07-24 19:00 | 已通过 | [详情] [签到] |
| [数据行] WH20260723... | 篮球赛   | 体育社 | 07-23 15:00 | 待审批 | [详情]       |
| ...                                                         |
+------------------------------------------------------------+
| 分页器                                                       |
+------------------------------------------------------------+
```

### 3.3 筛选栏设计

| 筛选项 | 组件 | 数据源 | 说明 |
|---|---|---|---|
| **状态** | 下拉多选 | 固定枚举 | 支持多选，默认「待审批+审批中」聚焦待办 |
| **主办社团** | 远程搜索下拉 | `GET /sys-club/list` | 仅显示用户有权限的社团（Level 2 仅本社团） |
| **活动类别** | 下拉单选 | `GET /activity-category/list` | 活动分类（如文艺演出、体育竞赛等） |
| **时间范围** | 日期选择器 | — | 按活动开始时间筛选 |
| **关键词搜索** | 输入框（防抖） | — | 按活动名称模糊搜索 |


## 4. 列表页 · 核心交互设计

### 4.1 列表列定义

| 列名 | 字段 | 展示规则 |
|---|---|---|
| **活动编号** | `activityNo` | 19位编号（如 `WH20260724143082739`），等宽字体，固定宽度 |
| **活动名称** | `activityName` | 链接样式，点击跳转详情页 |
| **主办社团** | `clubName` | — |
| **活动时间** | `startTime` ~ `endTime` | 格式 `MM-DD HH:mm` |
| **活动级别** | `activityLevel` | 1=院级（蓝色标签）、2=校级（金色标签） |
| **状态** | `approveStatus` | 见 §1.3 视觉标识 |
| **当前步骤** | `currentStep` / `approverName` | 显示“第X步/审批人姓名”（仅状态=2/3/7时显示） |
| **操作列** | — | 见 §4.2 |

### 4.2 操作列 · 动态渲染矩阵（列表级）

列表页的操作列**仅承载轻量操作**，所有复杂操作（审批、变更、总结）均收敛至详情页。

| 操作按钮 | 出现条件 | 点击行为 |
|---|---|---|
| **详情** | 所有用户（有列表页查看权限） | 跳转 `/activity/detail/:activityNo` |
| **去签到** | `当前用户在该活动中为签到执行人` + `活动状态=4` + `当前时间在签到窗口内` | 跳转 `/activity/sign/:activityNo` |
| **编辑** | `当前用户为发起人` + `活动状态=1（草稿）` | 弹出编辑抽屉（或跳转编辑页） |
| **继续提交** | `当前用户为发起人` + `活动状态=1（草稿）` | 确认提交，调用提交接口 |
| **撤回** | `当前用户为发起人` + `活动状态=2（待审批）` | 确认撤回，调用撤回接口 |

> **注意**：部长（Level 3）进入列表页时，操作列**仅有「详情」一个按钮**，无任何管理/发起能力。

### 4.3 「去签到」按钮的跳转逻辑

- 点击「去签到」跳转至 `/activity/sign/:activityNo`。
- 该页面为**轻量独立页**（无左侧菜单、无顶部导航），仅保留返回到活动列表的链接，确保活动现场快速签到。
- 页面内容：活动名称、签到时间、签到方式（定位/扫码）、签到按钮、签退按钮。


## 5. 详情页 · 完整信息与操作中心

### 5.1 页面布局

```
+------------------------------------------------------------+
| [返回列表] 活动名称                   状态标签   活动编号     |
+------------------------------------------------------------+
| 基本信息卡片                                                  |
|  主办社团 | 活动类别 | 活动级别 | 时间 | 地点 | 预算        |
|  活动内容描述                                                |
+------------------------------------------------------------+
| 审批链时间线（el-steps）                                     |
|  第1步: 社长审批  ✅ 已通过  张三  2026-07-23 10:00          |
|  第2步: 指导老师  ⏳ 待审批  李四  (当前)                    |
|  第3步: 学院书记  ⬜ 待处理  (未到)                          |
+------------------------------------------------------------+
| 操作区（依据身份 × 状态动态渲染）                            |
|  [通过] [驳回] [发起变更] [取消活动] [上传总结]              |
+------------------------------------------------------------+
| 签到管理面板（仅状态≥4时显示）                               |
|  签到配置 | 签到统计 | 签到记录列表                          |
+------------------------------------------------------------+
| 变更历史（折叠面板）                                         |
|  变更时间 | 变更字段 | 变更前 | 变更后 | 变更原因           |
+------------------------------------------------------------+
```

### 5.2 审批链时间线（核心组件）

- 使用 `el-steps` 垂直展示完整审批链。
- 每个步骤展示：角色名、审批人姓名、审批结果（通过/驳回）、审批意见、审批时间。
- 当前步骤特殊标识（金色高亮边框），并显示“等待中...”状态。
- 超时步骤（>3天）显示警告标识（黄色感叹号），>7天显示紧急标识（红色感叹号）。

### 5.3 操作区 · 详情级动态矩阵

> 下列操作按钮仅当**满足所有条件**时才可见，且必须由后端权限标识驱动。

| 操作 | 可见条件 | 交互行为 |
|---|---|---|
| **通过** | `当前用户为该步骤审批人` + `状态∈{2,3,7}` | 弹出审批意见对话框（必填），提交 `POST /activity-apply/approve/{id}`。审批人为指导老师时，可选择调整活动级别（仅一次机会） |
| **驳回** | `当前用户为该步骤审批人` + `状态∈{2,3,7}` | 弹出驳回意见对话框（必填），提交 `POST /activity-apply/reject/{id}`。变更审批驳回时，自动恢复原数据 |
| **发起变更** | `当前用户为发起人` + `状态=4（已通过）` | 弹出变更表单（仅允许修改时间/地点），提交后状态变更为 7（变更审批中） |
| **取消活动** | `当前用户为发起人` + `状态∈{2,3,4,7}` | 弹出二次确认框（需输入活动名称确认），提交后状态变更为 6（已取消） |
| **上传总结** | `当前用户为发起人` + `状态=4（已通过）` + `活动结束时间 ≤ 当前时间 ≤ 活动结束时间 + 3天` | 弹出总结对话框（文字内容 + 附件），提交 `POST /activity-apply/summary/{id}` |
| **查看签到统计** | `状态≥4` + 用户有签到统计查看权限 | 展开签到数据面板（签到率、迟到/早退分布、签到名单） |

### 5.4 状态=8（已封锁）的特殊处理

当活动因防篡改机制触发封锁时：
- 详情页顶部显示全宽红色警告条：**“该活动已被封锁，因检测到数据异常。请联系管理员。”**
- **所有操作按钮全部隐藏**（包括审批、变更、取消、签到）。
- 页面仅提供「返回列表」和「联系管理员」两个可交互元素。


## 6. 发起活动流程（仅社长/超管/指导老师）

### 6.1 入口

- 活动列表页右上角「发起活动」按钮。
- **可见性**：仅 `effectiveLevel ≤ 2` 可见（超管、院长、社长）。部长无此按钮。

### 6.2 表单字段（与后端 `ActivitySubmitDTO` 对齐）

| 字段 | 组件 | 必填 | 说明 |
|---|---|---|---|
| **主办社团** | 下拉选择 | ✅ | 仅显示用户有权限的社团（社长自动选本社团） |
| **活动名称** | 输入框 | ✅ | 最大 100 字符 |
| **活动类别** | 下拉选择 | ✅ | 从 `GET /activity-category/list` 获取 |
| **活动级别** | 单选（院级/校级） | ✅ | 院级=1，校级=2 |
| **开始时间** | 日期时间选择器 | ✅ | 必须早于结束时间 |
| **结束时间** | 日期时间选择器 | ✅ | — |
| **活动地点** | 输入框 | ✅ | — |
| **地点详情** | 输入框 | ❌ | — |
| **预计参与人数** | 数字输入 | ✅ | ≥ 1 |
| **活动预算** | 数字输入 | ✅ | ≥ 0 |
| **活动内容** | 富文本编辑器 | ✅ | — |
| **安全预案** | 富文本编辑器 | ✅ | — |
| **附件** | 文件上传 | ❌ | 支持多文件，单文件 ≤ 300MB |

### 6.3 提交行为

1. 点击提交 → 前端携带表单数据调用 `POST /activity-apply/submit`。
2. 后端生成 `activityNo`（19位编号），动态生成审批链并写入 `activity_approve_flow`。
3. 响应成功后：
   - 弹窗提示：**“活动申请已提交，审批编号：{activityNo}”**。
   - 列表页自动刷新，新增活动状态为「待审批」。
   - 若发起人=部长，因部长无发起权限，此流程不适用。


## 7. 审批流程（核心业务逻辑）

### 7.1 审批链动态生成规则（前端理解即可，无须实现）

审批链根据发起人角色自动生成，前端在详情页中展示即可。

| 发起人 | 审批步骤（按顺序） |
|---|---|
| 部长 | 社长 → 指导老师 → 学院书记 → 校书记（校级活动） |
| 社长 | 指导老师 → 学院书记 → 校书记（校级活动） |
| 指导老师 | 学院书记 → 校书记（校级活动） |
| 学院书记/管理员 | 指导老师 → 学院书记 → 校书记（校级活动） |

**院级活动**：终止于学院书记（不触发校书记）。
**校级活动**：完整链包含校书记。

### 7.2 超时机制（前端展示要求）

- 审批步骤进入后 **>3天** 未处理：步骤卡片显示**黄色警告** + 文字“该步骤已等待 3 天，建议催办”。
- 审批步骤进入后 **>7天** 未处理：步骤卡片显示**红色紧急警告** + 文字“该步骤已严重超时（>7天），系统已自动转交上级”。
- 超时信息从后端 `activity_approve_flow.step_enter_time` 计算得出，前端仅做展示。

### 7.3 审批操作详情

| 操作 | 触发条件 | 前端行为 |
|---|---|---|
| **通过** | 当前用户 = 该步骤审批人 | 弹窗：审批意见（选填，但后端要求必填，前端做非空校验）→ 若为指导老师且活动级别可调整 → 额外显示活动级别下拉 → 提交 |
| **驳回** | 当前用户 = 该步骤审批人 | 弹窗：驳回意见（必填）→ 提交。变更审批驳回时，后端自动恢复快照数据，前端不做额外处理 |

> **版本冲突处理**：提交时若后端返回 `ACT_VERSION_CONFLICT (7308)`，前端捕获后弹出提示“数据已被修改，请刷新页面重试”，并提供「刷新」按钮。


## 8. 签到执行页（轻量独立页）

### 8.1 页面定位

- 该页面为**无菜单、无导航的纯净页面**，适用于活动现场快速签到。
- URL：`/activity/sign/:activityNo`
- 进入方式：
  1. 列表页「去签到」按钮
  2. 活动详情页「签到」按钮
  3. **扫描签到二维码**（自动解析 `activityNo`） — 见 §8.4 二维码生成与扫码

### 8.3 签到操作

| 操作 | 触发条件 | 前端行为 |
|---|---|---|
| **定位签到** | `sign_mode` 包含定位 | 获取用户当前位置 → 计算与活动地点距离 → 若 ≤ 配置半径 → 调用 `POST /activity-sign/sign?activityNo=` 带经纬度 |
| **扫码签到** | `sign_mode` 包含扫码 | 打开摄像头扫描二维码 → 解析 `qrToken` → 调用 `POST /activity-sign/sign?activityNo=` 带 `qrToken` |
| **签退** | `checkoutEnabled=true` + 已签到 | 调用 `POST /activity-sign/checkout?activityNo=` |

### 8.4 二维码生成与扫码签到（扫码签到完整链路）

#### 8.4.1 二维码生成（管理后台）

**入口**：活动详情页 → 签到管理面板 → 「生成签到二维码」

| 交互 | 说明 |
|---|---|
| 权限 | 仅社长/指导老师/超管可生成（`effectiveLevel ≤ 2` 且为活动发起方） |
| 前置条件 | 活动状态 = 4（已通过）且签到配置已启用 `sign_mode` 包含扫码 |
| 点击行为 | 调用接口生成二维码 → 返回二维码图片 URL → 前端展示二维码，并提供「下载」按钮 |
| 有效期 | 二维码有效期为活动签到窗口的起始时间至结束时间（过期后扫码返回 6507） |
| 刷新机制 | 可手动刷新生成新二维码（旧码自动失效） |

**待实现接口**：

| 项 | 内容 |
|---|---|
| **路径** | `POST /activity-sign/generate-qr` |
| **请求参数** | `{ "activityNo": "WH20260724..." }` |
| **响应** | `{ "qrCode": "data:image/png;base64,...", "expireAt": "2026-07-24 20:00:00" }` |
| **说明** | 二维码中编码的 `qrToken` 应由后端生成（含有效期），前端仅展示返回的图片 |

#### 8.4.2 扫码签到（执行人）

| 项 | 内容 |

| **入口** | 签到执行页「扫码签到」按钮 |
| **交互流程** | 点击 → 打开摄像头 → 识别二维码 → 自动解析 `qrToken` → 调用签到接口完成签到 |
| **异常处理** | 二维码无效/过期 → 提示“二维码无效或已过期，请联系活动组织者” |

**待实现接口**（复用现有签到接口，但请求参数增加 `qrToken` 字段）：

| 项 | 内容 |
|---|---|
| **路径** | `POST /activity-sign/sign?activityNo={activityNo}`（复用现有接口） |
| **请求参数** | `{ "qrToken": "xxxxxx", "latitude": 39.9, "longitude": 116.4 }` |
| **响应** | 同现有签到接口响应 |
| **后端校验** | 校验 `qrToken` 合法性 → 校验有效期 → 校验当前用户是否有签到资格 → 校验是否在签到窗口内 → 记录签到 |

#### 8.4.3 二维码有效期管理

| 场景 | 前端行为 |
|---|---|
| **活动签到窗口开始前** | 不可生成二维码（按钮置灰，提示“签到尚未开始”） |
| **活动签到窗口内** | 可正常生成与扫码 |
| **活动签到窗口结束后** | 已有二维码失效，扫码返回 6507；不可重新生成 |
| **手动刷新二维码** | 旧二维码立即失效，新二维码生成（用于防止二维码被截图传播） |

| 6506（不在有效范围） | “您距离活动地点过远，请确认位置” |
| 6507（二维码无效/过期） | “二维码已过期或无效，请重新获取” |
| 6508（签到方式不支持） | “当前活动不支持该签到方式” |

## 9. 变更与取消

### 9.1 变更申请（仅状态=4 已通过）

- **入口**：活动详情页操作区「发起变更」。
- **约束**：
  - 仅允许修改 `startTime`、`endTime`、`location`。
  - 修改后提交，活动状态变为 7（变更审批中）。
  - 变更审批链：社长 → 指导老师 → 学院书记。
- **前端表单**：预填当前值，仅开放上述三个字段的编辑。
- **提交行为**：调用 `POST /activity-apply/change/{id}`，携带 `version` 乐观锁。

### 9.2 取消活动（仅发起人）

- **入口**：活动详情页操作区「取消活动」。
- **约束**：状态为 2/3/4/7 时可取消。
- **前端行为**：弹出二次确认框，**需手动输入活动名称**以确认。
- **提交行为**：调用 `POST /activity-apply/cancel/{id}`，携带 `version`。


## 10. 总结提交

### 10.1 时间窗口约束

- 仅在活动状态=4（已通过）且 `活动结束时间 ≤ 当前时间 ≤ 活动结束时间 + 3天` 时可用。
- 前端需**显式提示剩余时间**：“请在 X 天 X 小时内提交总结，逾期将无法提交。”

### 10.2 表单字段

| 字段 | 组件 | 必填 |
|---|---|---|
| 总结内容 | 富文本编辑器 | ✅ |
| 总结附件 | 文件上传（多文件） | ❌ |

### 10.3 提交行为

- 调用 `POST /activity-apply/summary/{id}`，携带 `version`。
- 提交成功后，活动详情页显示“总结已提交”，并展示总结内容与附件。
