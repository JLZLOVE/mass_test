> 文档性质：\*\*业务逻辑说明书与交互行为准则\*\*  

> 适用范围：`club-admin-frontend` 一级菜单「社团管理」

\## 1. 设计定位与用户分层



\### 1.1 功能域归属



社团是平台的核心组织单元，本模块服务于三类管理场景：



| 场景类型 | 用户层级 | 核心诉求 |

|---|---|---|

| \*\*全局监管\*\* | 超管、校级/院级管理员（Level 0/1） | 全平台社团检索、审批流程管控、异常解散干预 |

| \*\*社团自治\*\* | 社长（Level 2） | 本社团信息维护、发起解散申请、查看成员构成 |

| \*\*部门协作\*\* | 部长（Level 3） | 查看本部门挂靠社团信息、活动申请前置准备 |



> \*\*设计铁律\*\*：普通学生（Level 4）\*\*无本模块菜单入口\*\*，其社团数据消费需求通过\*\*前台门户\*\*、\*\*个人中心\*\*、\*\*活动详情页\*\*承载，详见 §1.3。



\### 1.2 页面级准入控制（路由层）



| 用户等级 | 左侧菜单可见性 | 页面访问权限 |

|---|---|---|

| Level 0/1（超管/院长） | ✅ 可见 | 全量数据 |

| Level 2（社长） | ✅ 可见 | 仅限本人管辖社团 |

| Level 3（部长） | ✅ 可见 | 仅限本部门关联社团 |

| \*\*Level 4（普通学生）\*\* | \*\*❌ 不可见（菜单不渲染）\*\* | \*\*无访问路径\*\* |



\### 1.3 Level 4 学生社团数据出口（设计补充）



普通学生不进入管理后台，但通过以下渠道获取社团信息：



| 出口位置 | 数据类型 | 权限控制 |

|---|---|---|

| \*\*前台门户（Portal）\*\* | 全量正常社团列表 + 详情 | 公开（`@IgnoreAuth`） |

| \*\*个人中心（Profile）\*\* | 当前学生所属社团缩略信息 | 仅限本人（`userId` 过滤） |

| \*\*活动详情页\*\* | 主办社团基本信息 | 公开（已通过审批的活动） |

\### 2.2 顶部筛选栏（数据驱动）



| 筛选项 | 组件类型 | 数据源 | 业务规则 |

|---|---|---|---|

| \*\*社团类别\*\* | 下拉选择框 | `GET /club-application/categories` | 禁止前端硬编码枚举，必须从接口动态获取。返回六类标准枚举（学术科技、文化体育、创新创业等），前端仅做纯渲染 |

| \*\*所属学院\*\* | 远程搜索下拉框 | `GET /sys-college/list` | 支持关键词检索，仅显示当前用户有权限查看的学院（Level 0/1 全量，Level 2/3 仅本学院） |

| \*\*状态\*\* | 下拉选择框 | 前端常量：`正常` / `已解散` | 与 Tab 联动，Tab 切换时自动重置该筛选项为“全部” |

| \*\*全局搜索\*\* | 输入框（防抖） | — | 占位符：`请输入社团名称或社团编号`。触发条件：输入停顿 ≥ 300ms 后自动检索 |



\### 2.3 顶部 Tab 切换栏（流程物理隔离 — 遵循 2B）



> \*\*设计意图\*\*：将“稳定态”与“流程态”彻底分离，避免表格内杂糅过多状态标签造成视觉干扰。



| Tab 名称 | 数据范围 | 业务说明 |

|---|---|---|

| \*\*正常社团\*\* | `status = 1`（正常运营） | 日常管理入口，承载查看详情、发起解散申请 |

| \*\*申请解散中\*\* | 关联 `club\_application` 表中 `apply\_type=2` 且 `status ∈ {1, 2}` | 跟踪解散审批进度，不可发起新申请 |

| \*\*合议中\*\* | 关联 `club\_council` 表中 `status = 1`（合议进行中） | 超管/校级专员签字入口 |



> \*\*状态联动约束\*\*：

> - 用户在“正常社团”Tab 下发起“解散申请”后，该社团应\*\*即时从当前列表消失\*\*，自动出现在“申请解散中”Tab，无需手动刷新。

> - 校级审批\*\*驳回\*\*解散申请（`club\_application.status=4`）时，该社团自动回退至“正常社团”Tab，并弹出驳回原因提示。

> - 合议签字达成条件并执行解散后，该社团自动移至“已解散”筛选范围。

\## 3. 列表数据展示规范



\### 3.1 列定义



| 列名 | 字段映射 | 展示规则 |

|---|---|---|

| \*\*社团编号\*\* | `clubCode` | 固定宽度，用于唯一标识。字体使用等宽字体增强辨识度 |

| \*\*社团名称\*\* | `clubName` | 支持点击跳转至详情页（权限见 §4） |

| \*\*社团类别\*\* | `category` + `categoryCode` | 显示后端返回的中文标签。列表左侧渲染 \*\*2px 色块\*\*：SZ（学术科技→科技蓝）、XS（学术科技→科技蓝）、CX（创新创业→活力橙）、WH（文化体育→香槟金）、GY（公益志愿→翠绿）、ZL（自律互助→紫色） |

| \*\*指导老师\*\* | `advisorName`（虚拟字段） | 若为 `null`，显示灰色占位“待指派” |

| \*\*成员总数\*\* | \*\*前端聚合计算（遵循 1A）\*\* | \*\*数据来源\*\*：并发请求 `GET /club-statistics/list?clubIds=\[...]` 聚合。\*\*性能规范\*\*：前端必须实现请求合并（Batch Request），一次携带当前页所有 `clubId` 查询，禁止循环单查。展示后缀“人”，空数据展示 `0人` |

| \*\*状态\*\* | `status` | 正常：绿色圆点 `●` + “正常”；已解散：灰色圆点 `●` + “已解散” |

| \*\*操作列\*\* | — | 依据 §4 权限矩阵动态渲染，\*\*严禁出现灰色禁用按钮\*\*（不可见即无权限） |



\### 3.2 空状态设计



当筛选条件下无数据时，展示插画空状态：

\- 文案：“当前暂无匹配社团”

\- 辅助建议：“请尝试调整筛选条件或清除搜索关键词”

\## 4. 操作列权限矩阵（行级细粒度控制）



> \*\*设计原则\*\*：操作按钮的显隐遵循 \*\*“可见即有权，不可见即无权”\*\*，严禁展示禁用态按钮。



| 操作名称 | 图标样式 | 可见性规则（逻辑与） | 交互结果（遵循 3B） |

|---|---|---|---|

| \*\*查看详情\*\* | 文本链接 / 眼睛图标 | 当前用户的 `effectiveLevel ≤ 3`（超管、院长、社长、部长） | 点击路由跳转至 `/club/detail/{clubCode}` 独立详情页。详情页承载：社团信息卡、部门架构树、成员列表（分页）、历史活动时间线。跳转携带 `from=club-list` 参数用于面包屑回退 |

| \*\*解散申请\*\* | 文本链接（警示色） | \*\*条件 1\*\*：`effectiveLevel ≤ 2`（超管、院长、社长）<br>\*\*条件 2\*\*：当前用户是该社团绑定社长（`scope\_type=2` 匹配）或指导老师（`advisor\_id` 匹配）<br>\*\*条件 3\*\*：社团 `status = 1`（正常） | 点击弹出二次确认对话框（需输入社团名称二次确认），调用 `POST /club-application/apply/dissolve`（Body 传 `clubCode`）。成功后\*\*自动切换 Tab 至“申请解散中”\*\*，并弹出轻提示“解散申请已提交，等待审批” |

| \*\*合议签字\*\* | 文本链接（强调色） | `effectiveLevel ≤ 1` 且拥有校级管理权限（`ROLE\_ADMIN` 或 `SUPER\_ADMIN`），且社团 `status = 1` | 点击跳转至 `/club/council/{clubId}` 独立详情页，展示合议签字面板（包含签字人列表、签字状态、签字按钮） |



> \*\*⚠️ 禁止事项\*\*：严禁在前端使用 `v-if="role === '社长'"` 硬编码字符串判断。必须使用后端下发的权限标识（如 `canDissolve: boolean`）或基于 `scopeId` 与 `clubId` 的运行时比对。

\## 5. 关键交互闭环设计



\### 5.1 成员数聚合策略（遵循 1A）



| 设计维度 | 规范说明 |

|---|---|

| \*\*数据来源\*\* | 前端主动并发请求 `GET /club-statistics/list`，携带当前页所有 `clubId` 列表 |

| \*\*请求优化\*\* | 使用 `Promise.all` 或 `Promise.allSettled` 批量请求，单个社团统计接口失败时展示 `--` 占位符，不影响列表整体渲染 |

| \*\*刷新时机\*\* | ① 页面首次加载 ② 切换 Tab 时 ③ 用户从“成员管理”子页面返回时（`onActivated` 钩子） |

| \*\*缓存策略\*\* | 前端维护短时内存缓存（TTL 5秒），防止列表页与详情页间反复切换触发无意义重复请求 |

| \*\*加载状态\*\* | 成员数列展示骨架屏（Skeleton），数字出现前不展示 `0人` 误导信息 |



\### 5.2 列表与流程的状态联动（闭环）



| 触发事件 | 前端响应 |

|---|---|

| \*\*发起解散申请成功\*\* | ① 当前社团从“正常社团”Tab 移除 ② 自动切换至“申请解散中”Tab ③ 弹窗提示“申请已提交，审批编号：SQ\*\*\*\*\*” |

| \*\*解散申请被驳回\*\* | ① 社团回退至“正常社团”Tab ② 顶部出现全局 Message 横幅（持续 5 秒）：“驳回原因：{opinion}” |

| \*\*合议签字完成且解散执行\*\* | ① 社团从“合议中”Tab 移除 ② 跳转至“正常社团”Tab（此时该社团状态已变更为已解散，不再出现） ③ 弹窗提示“社团已解散” |

| \*\*校级管理员主动干预\*\* | 超管可在“申请解散中”Tab 直接驳回（无需跳转详情页），驳回后同上处理 |



\### 5.3 详情页跳转契约（遵循 3B）



| 契约项 | 规范 |

|---|---|

| \*\*路由路径\*\* | `/club/detail/:clubCode`（使用 `clubCode` 而非 `id`，对齐后端对外暴露规范） |

| \*\*携带参数\*\* | `from=club-list` 用于详情页面包屑正确回退 |

| \*\*详情页内容\*\* | ① 社团信息卡（名称、编号、类别、挂靠学院、指导老师、创建时间） ② 部门架构树（`sys\_department`） ③ 成员列表（分页，支持按角色筛选） ④ 历史活动时间线（关联 `activity\_apply`） |

| \*\*返回行为\*\* | 点击面包屑“社团管理”返回列表页时，\*\*保持当前 Tab 和筛选条件不变\*\*（使用 `keep-alive` + 路由缓存） |

\## 6. 异常与边界状态处理



| 异常场景 | 前端表现 |

|---|---|

| \*\*空状态\*\* | 展示插画空状态 + 文案“当前暂无匹配社团” + 辅助建议“请尝试调整筛选条件” |

| \*\*聚合接口超时/失败\*\* | 成员数列显示 `--` 灰色占位符，右侧暴露“重试”图标按钮（仅针对该行数据，不刷新全页） |

| \*\*并发解散冲突\*\* | 后端返回 `CLUB\_APPLY\_STATUS\_INVALID (7207)`，前端捕获后弹窗提示“该社团已有进行中的解散流程，请勿重复操作”，并自动刷新列表 |

| \*\*无权限访问详情页\*\* | 路由守卫拦截，重定向至社团列表页，并弹出 `ElMessage.warning("您无权查看该社团详情")` |

| \*\*网络断线\*\* | 列表页顶部出现全局断网提示条（不可关闭），网络恢复后自动重试当前请求 |

\## 7. 设计验收标准（Checklist）



\- \[x] 社团类别下拉框数据完全依赖后端接口，无前端枚举硬编码。

\- \[x] 操作列严格执行“部长以上可见详情，社长/指导老师起可见解散”的权限映射，且基于后端下发的权限标识符驱动。

\- \[x] 顶部 Tab 将“正常”、“解散中”、“合议中”物理隔离，列表数据互不污染。

\- \[x] 成员数采用前端并发聚合方案，但聚合请求入口统一封装（`useMemberCount`），避免冗余散落代码。

\- \[x] 所有涉及社团解散、合议签字的操作，均拒绝在列表页弹窗直接执行，强制跳转详情页（遵循 3B 与 2A 连贯性）。

\- \[x] Level 4 学生无菜单入口，其社团数据消费需求通过门户/个人中心/活动详情页承载，设计稿中已明确标注。

\- \[x] 空状态、加载状态、异常状态均已定义，无白屏或崩溃风险。

\## 8. 附录：相关接口依赖清单



> 供后端确认接口字段是否满足设计需求



| 用途 | 接口路径 | 关键字段要求 |

|---|---|---|

| 获取社团列表 | `GET /sys-club/list` | 必须返回：`clubCode`, `clubName`, `category`, `categoryCode`, `advisorName`（虚拟字段）, `status` |

| 获取社团类别枚举 | `GET /club-application/categories` | 返回 `{ code: "SZ", label: "学术科技" }` 结构 |

| 获取学院列表 | `GET /sys-college/list` | 返回 `id`, `collegeName` |

| 批量获取成员数 | `GET /club-statistics/list` | 请求参数：`clubIds`（逗号分隔），返回 `{ clubId, memberCount }` |

| 发起解散申请 | `POST /club-application/apply/dissolve` | Body：`clubCode`, `dissolveReason` |

| 创建社团 | `POST /club-application/apply/create` | 仅超管/院长可见，Body：`clubName`, `collegeId`, `category`, `proposedLeaderUsername`, `maxMembers` |
