

## 通知模块完整逻辑

### 一、发送通知

#### 入参

| 字段         | 必填 | 说明                                              |
| ------------ | ---- | ------------------------------------------------- |
| 标题         | 是   | 通知标题                                          |
| 内容         | 是   | 富文本内容                                        |
| 分类ID       | 是   | 关联 `notice_category`                            |
| 接收范围类型 | 是   | 1全体学生 2全体老师 3指定角色 4指定社团 5指定人员 |
| 接收范围值   | 是   | JSON格式，对应接收范围类型的ID列表                |
| 重要程度     | 否   | 3低 2中 1高                                       |
| 紧急程度     | 否   | 2不紧急 1紧急                                     |
| 是否需要确认 | 是   | true/false                                        |
| 定时发送时间 | 否   | 为空则立即发送                                    |
| 是否置顶     | 否   | true/false                                        |
| 置顶结束时间 | 否   | 置顶时必填                                        |
| 是否长期可见 | 是   | true=新成员也能看，false=仅当前成员               |
| 附件         | 否   | 支持多个附件                                      |
| 附件权限     | 否   | 最小等级要求                                      |

#### 权限校验

| 发送方                | 默认范围   | 跨范围需求                                                   |
| --------------------- | ---------- | ------------------------------------------------------------ |
| 管理员                | 全校       | 不需审批                                                     |
| 校书记（SUPER_ADMIN） | 全校       | 向其他学院发需目标学院书记同意 + 申请老师同意                |
| 指导老师              | 所指导社团 | 向其他社团发需本院书记审批；跨学院需两院书记 + 申请老师三方同意 |
| 社长                  | 本社团     | 不需审批                                                     |
| 部长                  | 本部门     | 发全社团需社长授权                                           |

#### 接收人固化规则

- **长期可见=false**：发布时写入 `notice_read_record`，记录当前所有接收人；后续新增人员看不到该通知
- **长期可见=true**：发布时只存 `receiver_values`，查询时动态计算接收人（新增成员也能看到）

**"长期可见"由发起人决定**：社长发本社团通知 → 社长确定；指导老师发所指导社团通知 → 指导老师确定；书记发全校通知 → 书记确定。

**可撤销性**：发起人可撤回通知，撤回后该通知的"长期可见"失效。

### 二、重要程度与紧急程度

| 字段     | 值   | 含义                     |
| -------- | ---- | ------------------------ |
| 重要程度 | 1    | 高（置顶展示、加红标识） |
| 重要程度 | 2    | 中（置顶展示）           |
| 重要程度 | 3    | 低（正常展示）           |
| 紧急程度 | 1    | 紧急（推送+置顶+加红）   |
| 紧急程度 | 2    | 不紧急（正常展示）       |

重要程度+紧急程度组合规则：两者任一为1时，通知置顶展示并加红标识。

### 三、附件权限

附件存储在本地 `uploads/notice/`，预留 OSS 接口。

附件权限规则：发送时指定 `attachment_min_level`（数字越小门槛越高，0仅超管可下载，4全体可下载）。用户查看通知时，若 `当前用户等级 <= attachment_min_level`，显示附件下载按钮；否则附件隐藏。

### 四、通知模板

独立接口：新增/更新/删除/列表/详情

删除模板时检查是否被通知引用，被引用则阻止删除，改为 `status=0`（停用，内容保留）。

### 五、通知撤回

- **发起人**：可撤回自己发送的通知（管理员可撤回全校，社长可撤回本社团，书记可撤回自己发的）
- **动作**：`status=2`（已撤回）
- **可见性**：等级 ≥ 1（管理员及以上）仍可见，其余人员不显示
- **已读统计**：保留历史数据，不删除
- **长期可见**：撤回后失效

### 六、置顶定时结束

定时任务每分钟扫描：`is_pinned=1` 且 `pin_expire_at < NOW()` → `is_pinned=0`

### 七、定时发送

定时任务每分钟扫描：`status=0`（草稿）且 `publish_time <= NOW()` → 发送并更新 `status=1`

### 八、通知草稿

`status=0` 草稿状态，前端可保存但不发送。定时发送时间有值时，定时任务到时自动发送；无值时，用户手动发送时触发。

### 九、已读/确认

- 用户进入通知详情页即记录 `read_time`（已读）
- 需要确认的通知（`need_confirm=true`），用户点击"确认阅读"后记录 `is_confirmed=1`、`confirm_time`
- 已读 ≠ 已确认，确认是已读的子集
- 发布人可查看已读人数和已确认人数

### 十、活动取消时的自动通知

活动取消时自动发布通知，内容为"【活动取消通知】XXX活动已取消"，接收范围为该活动的所有参与人。该通知由系统自动生成，发起人为活动申请人，不可撤回。
